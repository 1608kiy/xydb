# 03 技术方案与架构设计

## 1. 技术选型建议

推荐：Flutter（Dart）

原因：

- iOS/Android 一套代码，UI 还原能力强，适合“与 WEB 一摸一样”的交付目标。
- 状态管理与路由生态成熟（Riverpod/Bloc + go_router）。
- 图表能力完善（fl_chart/echarts_flutter）。

可选：React Native（TypeScript）

- 如果团队 JS/TS 经验明显更强，可采用 RN + Zustand/Redux Toolkit。

## 2. 客户端架构

建议分层：

- `presentation`：页面与组件。
- `application`：用例（登录、建任务、打卡等）。
- `domain`：实体与业务规则。
- `infrastructure`：API Client、本地存储、日志、网络。

建议目录：

- `lib/app`：启动、路由、主题。
- `lib/modules/auth`
- `lib/modules/tasks`
- `lib/modules/calendar`
- `lib/modules/pomodoro`
- `lib/modules/checkin`
- `lib/modules/reports`
- `lib/modules/profile`
- `lib/shared`：网络、错误码、组件库。

## 3. 状态管理与数据策略

- 全局状态：用户会话、主题、基础配置。
- 业务状态：按模块隔离（任务/番茄/打卡/报表）。
- 本地缓存：
  - Token：安全存储（Keychain/Keystore）。
  - 最近数据：SQLite/Isar（可选）。
- 网络策略：
  - 先拉后显 + 下拉刷新。
  - 写操作统一走远端成功回写。

## 4. API 网关与网络层约束

- 统一 Base URL 配置（dev/staging/prod）。
- 统一请求头注入（Authorization）。
- 统一错误处理：401 自动清会话并跳登录。
- 统一重试策略：幂等请求可重试一次。

## 5. 安全与合规

- JWT 不落明文普通存储。
- 敏感日志脱敏（邮箱、手机号、token）。
- 发布包开启混淆（R8/Proguard，iOS strip symbols）。

## 6. 性能指标建议

- 首屏可交互 < 2.5s（中端 Android 机型）。
- 页面切换平均 < 300ms。
- 任务列表 500 条滚动无明显掉帧。
