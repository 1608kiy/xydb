# 项目定制化测试流程

## 项目背景
本项目为面向个人效率管理的全栈应用，核心功能包括：
- 任务管理（增删改查、优先级、标签、子任务、筛选排序）
- 日历排程（月/周/日视图、AI 智能排程）
- 番茄钟专注（会话记录与统计）
- 打卡习惯（日历打卡、最近打卡）
- 数据周报（统计概览、AI 建议 + 兜底）
- 账号设置（登录/注册/忘记密码/用户信息）

技术栈：前端 MPA + Tailwind CDN + 原生 JS（common.js 统一管理），后端 Spring Boot + Spring Security + JPA + JWT，数据库支持 MySQL（生产）与 H2（本地联调）。最新迭代引入后端 AI 代理接口（`AiController`），前端优先调用后端代理，失败时回退直连，并增加了多处兜底机制。

---

## 1. 测试目标
- **功能完整性**：验证所有产品能力按预期工作，核心业务流程无阻塞。
- **AI 集成稳定性**：确保 AI 代理接口正常工作，兜底机制生效，用户体验不中断。
- **安全性**：消除前端密钥暴露、越权访问等风险，确保 JWT 认证正确。
- **兼容性**：在主流浏览器及常用分辨率下界面正常、交互流畅。
- **数据正确性**：前后端数据一致，数据库操作符合业务逻辑。
- **可维护性**：确认项目无噪音文件污染，日志与临时文件可控。

## 2. 测试准备

### 2.1 环境准备
- **开发环境**：本地运行后端（H2 内存库）及前端（直接打开 HTML 文件或使用 live server），用于快速验证。
- **测试环境**：部署到云服务器（或本地虚拟机），使用 MySQL 数据库，模拟真实生产环境。
- **工具**：
  - 浏览器：Chrome（最新版）、Firefox（最新版）、Safari（若需）
  - Postman / Insomnia：测试 API
  - 数据库客户端：DBeaver、MySQL Workbench 或命令行
  - 抓包工具：浏览器开发者工具、Charles（可选）

### 2.2 数据准备
- 准备多组测试账号：
  - 普通用户 A（有完整数据）
  - 普通用户 B（无数据）
  - 管理员账号（若支持）
- 准备测试数据：若干任务、日历事件、番茄钟会话、打卡记录，确保覆盖各种状态（已完成、未完成、优先级高中低、标签等）。

### 2.3 代码与配置检查
- 确认 `application.properties` 或 `application.yml` 中生产环境配置正确（MySQL 连接、JWT 密钥、AI 代理地址等）。
- 确认前端 `common.js` 中不再保留 AI 默认密钥（安全收口）。
- 确认 `.gitignore` 已包含 `backend_run.log`、`backend_run.err`、`REPORTS/`、`tmp/` 等噪音文件。

## 4. AI 集成专项测试

### 4.1 后端 AI 代理接口（`AiController`）测试
- **正常请求**：使用 Postman 调用代理接口，传入合法的提示词，验证返回结果格式正确。
- **异常处理**：
  - 请求超时（模拟 AI 服务响应慢）：后端应返回超时错误码，前端捕获后回退直连。
  - AI 服务返回错误（如 5xx）：后端应包装错误响应，前端进入兜底。
  - 缺少必填参数：后端返回 400。
- **认证**：确认代理接口需要 JWT token 认证，未携带 token 返回 401。

### 4.2 前端 AI 调用链路
- **优先级**：前端代码应首先调用后端代理接口，仅当失败（网络错误、后端返回错误码）时才调用直连。
- **直连回退**：模拟后端代理不可用（如停止后端服务），观察前端是否自动切换到直连，且功能正常（注意直连的 API key 应已移除，需临时使用测试 key）。
- **兜底文案**：在 AI 建议失败时，前端应展示预设的默认建议，且不抛出控制台错误。

### 4.3 AI 排程异步优化
- **并发测试**：快速连续点击多个任务的 AI 排程，观察请求队列处理，避免重复覆盖。
- **状态反馈**：AI 优化完成时，日历上应有明确的提示（如 toast 消息）告知用户结果。

## 5. 安全测试（重点）

### 5.1 密钥与敏感信息
- 检查前端代码（`common.js`、各页面内联脚本）中是否仍存在 AI 密钥、JWT 密钥等明文。
- 确认生产环境的后端配置文件中不包含任何明文密码，使用环境变量或密钥管理服务。

### 5.2 认证与授权
- **未登录访问**：直接访问需要认证的页面或 API，应跳转登录或返回 401。
- **越权访问**：
  - 用户 A 登录，尝试获取用户 B 的任务列表（通过修改 API 路径或参数），应返回 403 或空数据。
  - 若有管理员功能，普通用户 token 调用管理员接口应返回 403。
- **JWT 有效期**：设置合理过期时间，过期后需重新登录。

### 5.3 输入验证
- 前端对输入做了基本校验，但后端仍需进行二次校验，防止恶意数据入库。
- 测试特殊字符、SQL 注入（如 `' OR '1'='1`）在登录、搜索等接口的表现，应被转义或拒绝。

## 7. 数据一致性测试

- **事务测试**：模拟创建任务时同时写入子任务，若子任务插入失败，父任务不应保存（可通过后端日志确认）。
- **并发测试**：同时使用两个浏览器窗口对同一任务进行编辑，后提交的应基于最新数据（乐观锁或提示冲突）。
- **数据库直接操作**：通过 SQL 修改数据后，刷新前端页面，应看到更新后的数据。

## 8. 性能与稳定性测试（简化）

由于是个人项目，无需专业压测，但需关注：
- **首屏加载**：使用浏览器 Network 标签，查看页面加载时间，静态资源应开启缓存。
- **AI 接口响应**：AI 代理接口应设置合理超时（如 10 秒），避免长时间阻塞页面。
- **日志监控**：部署后定期检查 `backend_run.log`，确认无频繁错误堆栈。

## 9. 噪音文件与工程化检查

- 确认 `.gitignore` 已忽略：`*.log`、`REPORTS/`、`tmp/`、`backend_run.*`、`.DS_Store` 等。
- 运行 `git status`，当前工作区应仅包含必要的源代码变更，无临时文件。
- 若存在 `DEVELOPMENT.md` 或 `README.md` 中提及的脚本，验证其可用性（如 `./run-dev.sh` 能正常启动 H2 联调环境）。

## 10. 缺陷管理与回归

- 使用 GitHub Issues 或简单 Excel 记录发现的问题，按严重程度（P0/P1/P2）分级：
  - **P0**：核心功能完全不可用、数据丢失、严重安全漏洞（如密钥泄露）。
  - **P1**：非核心功能失效、体验严重受阻、兼容性问题。
  - **P2**：文案错误、样式瑕疵、优化建议。
- 修复后，对相关模块执行回归测试，确保未引入新问题。
