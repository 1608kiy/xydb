# XYDB 全链路测试计划（软件端 + 网页端 + 前后端连通 + APK 真机）

## 1. 文档信息
- 文档版本：v1.1
- 编写日期：2026-04-13
- 项目名称：XYDB（铃记）
- 适用范围：
  - 软件端：SFOTWARE/frontend（后续打包 APK）
  - 网页端：WEB/frontend
  - 后端：WEB/backend
  - 共享模块：SHARE/common.js、SHARE/data.js
- 测试目标：为后续服务器部署提供完整、可执行、可验收的测试依据。

## 1.1 本次补强说明（v1.1）
- 新增阶段准入/准出标准，避免测试范围大但无法判定是否可发布。
- 新增测试数据治理与清理机制，避免脏数据干扰结果。
- 新增数据库迁移与备份恢复测试，补齐部署前高风险环节。
- 新增生产可观测性与告警演练项，确保线上问题可发现可定位。
- 新增兼容性与可访问性条目，完善“全面测试”覆盖面。

## 2. 测试目标与成功标准
### 2.1 总体目标
- 验证核心业务功能可用且稳定。
- 验证网页端与软件端在同一账号下数据一致。
- 验证前端与后端接口联通、异常可恢复。
- 验证 APK 打包后在 Android 真机可稳定运行。
- 验证上线前具备可观测性、可回滚性与最小运维风险。

### 2.2 成功标准（上线门禁）
- P0/P1 缺陷清零。
- P2 缺陷有明确规避方案且不影响主流程。
- 关键接口可用率 >= 99.5%（测试窗口内）。
- 登录成功率 >= 98%。
- 核心页面首屏加载时间：
  - Wi-Fi：<= 2s
  - 4G/5G：<= 3s
- 最小验收套件与毕业级套件全部通过。
- APK 真机专项通过（安装/升级/弱网/后台恢复/兼容性）。

## 3. 测试范围
### 3.1 功能范围（必须覆盖）
- 账号与认证：注册、登录、登出、找回密码、管理员识别。
- 待办域：任务增删改查、子任务、标签、筛选、排序、完成状态。
- 日历域：月/周/日切换、任务排程展示、筛选。
- 番茄域：模式切换、计时、暂停/恢复、重置、记录入库。
- 打卡域：打卡创建、最近记录、日历标记、统计。
- 周报域：总览、趋势、分类、时间段分析、建议。
- 个人中心：资料读取与更新、安全设置、退出登录。
- 管理域（Web）：用户列表、删除、批量删除、创建管理员。
- AI（Web）：对话接口可用性与鉴权。

### 3.2 链路范围（必须覆盖）
- 前端到后端 API 联调。
- 软件端到后端 API 联调。
- 软件端与网页端跨端一致性。
- 弱网/断网/恢复网络下数据一致性与重试表现。
- 部署前后冒烟验证（本地、预发布、生产）。

### 3.3 非目标（本轮不做）
- 多租户复杂权限模型。
- 聊天 AI 多轮长期上下文质量评测。
- 大规模性能压测（10k+ 并发），仅做容量前置评估。

## 4. 测试环境与矩阵
### 4.1 环境分层
- DEV：开发联调环境（本地 local profile）。
- STAGING：预发布环境（尽量贴近生产配置）。
- PROD-SMOKE：生产灰度后冒烟验证。

### 4.2 终端矩阵

  - Chrome（最新稳定版）
  - Edge（最新稳定版）
- Android 真机（APK）：
  - Android 10/12/14 各至少 1 台
  - 分辨率覆盖：720p、1080p
  - 网络覆盖：Wi-Fi、4G/5G、弱网
- 可选补充：iOS Safari（H5 兼容观察）

### 4.3 后端与依赖
- JDK 17+
- Maven Wrapper（WEB/backend/mvnw.cmd）
- MySQL 8（生产路径）
- local profile 可使用 H2 联调

### 4.4 测试账号
- 普通用户账号：2 个（A/B）
- 管理员账号：1 个
- 新注册账号：按用例动态生成

### 4.5 测试数据治理
- 数据分层：
  - 基线数据：固定账号、固定标签、固定任务模板。
  - 动态数据：自动化执行时实时创建（带时间戳）。
  - 回归数据：用于趋势对比，禁止覆盖。
- 数据隔离：
  - DEV/STAGING/PROD-SMOKE 使用独立数据库或独立 schema。
  - 自动化测试账号统一前缀（如 auto-、accept-），便于清理。
- 数据清理：
  - 每轮回归后清理动态测试数据，保留报告与日志。
  - 每周执行一次全量测试账号巡检，清理陈旧数据。
- 数据保护：
  - 生产禁止使用真实用户数据进行破坏性测试。
  - 导出样例用于分析时必须脱敏（邮箱、手机号、token）。

## 5. 测试策略
### 5.1 分层策略
- L1 单元/组件级：工具函数、数据转换、页面关键方法。
- L2 接口级：认证、任务、番茄、打卡、报表、管理、AI。
- L3 集成级：页面交互 + API + 本地缓存。
- L4 端到端级：跨页面、跨端、跨网络场景。
- L5 发布级：回归套件 + 真机验收 + 部署后冒烟。

### 5.2 覆盖策略
- 按风险优先：认证/任务/同步/报表链路优先。
- 按业务优先：P0 页面六大核心页必须全覆盖。
- 按异常优先：弱网、超时、401/403/5xx、接口字段兼容。

### 5.3 阶段准入/准出（Entry/Exit Criteria）
- 准入条件（进入完整回归前必须满足）：
  - 后端健康检查通过，关键接口可达。
  - 测试环境版本冻结（前端 commit、后端 commit、配置版本可追溯）。
  - 测试账号可用，数据库连接正常。
  - 自动化脚本可执行，报告目录可写入。
- 准出条件（阶段完成判定）：
  - 冒烟阶段：P0 链路全部通过，无阻断缺陷。
  - 完整回归阶段：P0/P1 用例通过率 100%，P2 有结论。
  - 真机专项阶段：安装升级、弱网、前后台恢复全部通过。
  - 发布门禁阶段：自动化 PASS + 人工高风险抽检通过 + 回滚预案可执行。

### 5.4 需求追踪矩阵（RTM）
- 每条 P0/P1 需求必须映射到：
  - 至少 1 条功能用例（手工）
  - 至少 1 条接口用例（自动或手工）
  - 至少 1 条发布门禁检查项
- 追踪维度建议：需求 ID、页面/接口、用例 ID、自动化脚本、最近结果、缺陷单号。
- 发布评审时必须出具 RTM，确保“需求无遗漏、用例可追溯、风险有归属”。

## 8. 自动化执行方案
## 8.1 现有脚本（优先复用）
在 WEB 目录执行：
- scripts/verify_api.ps1：接口验证 + 测试任务创建。
- scripts/run_min_acceptance.ps1：最小验收 5 用例。

## 8.2 建议执行节奏
- 每次提交前：verify_api + 最小验收。
- 每日构建：回归 + UI 自动化。
- 发布前：毕业级套件 + 真机专项 + 人工探索回归。

## 8.3 报告产物
- 统一输出目录：WEB/REPORTS/e2e
- 产物类型：JSON + Markdown + 日志文件
- 要求：每次测试保留时间戳，支持回溯。

### 8.4 质量度量看板（建议）
- 用例通过率：按 P0/P1/P2 分层统计。
- 缺陷趋势：新增/关闭/遗留数量按天统计。
- 接口质量：成功率、P95 延迟、5xx 比例。
- 跨端一致性：双端一致通过率。
- 真机专项：机型通过率、系统版本通过率。

### 8.5 测试证据与审计要求
- 每个关键结论必须有证据：
  - 自动化报告（JSON/Markdown）
  - 关键步骤截图或录屏（真机优先录屏）
  - 失败日志与修复后复测记录
- 证据命名规范建议：
  - 模块_用例ID_环境_日期_结果
  - 示例：todo_B01_staging_20260413_PASS
- 证据保留周期：
  - 发布版本至少保留 2 个迭代周期。
  - 重大故障相关证据长期归档。

## 9. 测试执行流程（建议）
1. 环境准备
- 启动后端 local profile，确认健康检查通过。
- 准备普通账号/管理员账号。

2. 冒烟阶段（30~60 分钟）
- 登录、任务创建、番茄、打卡、周报查询。
- 通过后进入完整回归。

3. 完整回归阶段（0.5~1 天）
- 执行功能回归、接口合同、异常恢复、跨端一致性。
- 同步记录缺陷并回归验证修复。

4. 真机专项阶段（0.5 天）
- APK 安装升级、弱网、后台恢复、兼容性。

5. 发布前门禁（30 分钟）
- 执行自动化套件，确认报告 PASS。
- 人工抽检 10 条高风险用例。

## 10. 缺陷管理与优先级
- P0：阻断主流程/数据错误/无法登录/无法同步。
- P1：核心功能可绕过但影响明显。
- P2：次要功能问题，不影响核心链路。
- P3：UI 文案或低风险体验问题。

处理时效建议：
- P0：当天修复并回归。
- P1：48 小时内修复并回归。
- P2/P3：排期修复。

## 11. 角色分工建议
- 测试负责人：维护测试计划、门禁结论、上线建议。
- 前端负责人：页面功能与跨端一致性修复。
- 后端负责人：接口稳定性、权限、性能与日志。
- 运维/部署负责人：环境可用性、监控告警、回滚演练。

## 12. 风险与预案
- 风险 1：接口字段变更导致前端兼容问题。
  - 预案：接口契约冻结 + 变更评审 + 向后兼容。
- 风险 2：弱网下同步失败造成数据不一致。
  - 预案：离线队列 + 重试 + 状态可视化。
- 风险 3：APK 真机表现与浏览器调试不一致。
  - 预案：提前真机回归，不把问题留到临发布。
- 风险 4：上线后异常不可观测。
  - 预案：健康检查、关键指标告警、日志留存。
